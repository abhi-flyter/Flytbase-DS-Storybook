01 / What this is

Three things work together.

Storybook is where humans inspect the component library. MCP is how agents read the same component documentation before writing code.

Storybook

Open components, examples, props, usage notes, and visual states in the browser.

Chromatic

Hosts the latest published Storybook so teammates do not need to run it locally.

Storybook MCP

Lets agents query documented components and props before they build UI.

04 / Agent rules

Put this in the product repo instructions.

Add this behavior to AGENTS.md or CLAUDE.md so agents do not invent component props. 

When working on UI components, always use fb-design-system-sb-mcp before answering or taking action.

Never hallucinate component properties.

First list documented components.
Then read the documentation for the component you plan to use.
Only use props and examples that are explicitly documented.
If a prop is missing, ask for confirmation instead of guessing.
05 / Daily workflow

How an agent should build a screen.

The agent should inspect docs first, choose components second, code third, and verify at the end.

1
Discover
Ask MCP for the available documented components.
2
Inspect
Read docs for Button, TextField, InputField, Table, Tabs, and any component being used.
3
Compose
Build the screen using documented props and examples.
4
Verify
Check Storybook docs, run local tests when available, and review the UI in browser.
06 / Good prompt

Give agents a clear contract.

A good prompt names the product screen, requires MCP lookup, blocks undocumented props, and asks for verification. 

Use fb-design-system-sb-mcp before building.

Create a Flytbase drone operations settings page using only documented Flytbase design system components.

Before coding:
1. List the components you found in Storybook.
2. Read the docs for each component you plan to use.
3. Tell me if any needed interaction prop is not documented.

When coding:
- Use documented props only.
- Do not use native select, range, or number controls if the design system has a documented component.
- Verify the page in the browser before finishing.
07 / What to avoid

Most failures come from guessing.

These are the patterns that lead to UI that looks close but drifts away from Figma and Storybook.

Do not

  • Assume props such as shadow, variant, or onChange exist.
  • Use browser-native dropdowns when InputField exists.
  • Use browser-native range inputs when Slider exists.
  • Copy styles from memory instead of using documented components.

Do

  • Query Storybook docs first.
  • Use documented props and examples.
  • Ask when interaction docs are missing.
  • Verify in the browser before sharing output.
08 / Human review

What you still need to check manually.

MCP helps agents avoid hallucination. It does not replace design judgment or visual review.

Agent checks Component list, prop docs, examples, generated code, local render.
Developer checks Typecheck, Storybook build, tests, browser smoke, code review.
Designer checks Figma parity, spacing, hierarchy, states, interaction quality.
Product checks Does the screen solve the workflow clearly for the operator?
09 / Limits

What remote MCP can and cannot do.

Set expectations clearly for teammates so they do not think the remote MCP is broken.

Remote MCP can

  • Expose published Storybook documentation.
  • Help agents find components and documented props.
  • Reduce design-system hallucinations.

Remote MCP cannot

  • Run local story tests in your repo.
  • Guarantee Figma pixel parity by itself.
  • Know undocumented product-specific behavior.
10 / Ready checklist

Before sharing agent-built UI.

Use this checklist before a teammate treats the result as design-system compliant.

Agent evidence

The agent says which Storybook components and docs it used.

No undocumented props

Every design-system prop in code is documented or shown in a story example.

No native replacements

Dropdowns, sliders, toggles, tabs, tables, and form controls use the design system.

Human review complete

Designer or owner checks visual parity and product workflow quality.

Use left/right arrow keys, Space, or the buttons.
